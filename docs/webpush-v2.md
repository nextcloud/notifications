# Web push notifications

## Introduction

> Why is web push different from the [mobile push notifications](push-v2.md)?

The [mobile push setup](push-v2.md) relies on the `push-notifications.nextcloud.com` proxy because notifications have to be signed with Nextcloud's Google/Apple developer keys before *Firebase Cloud Messaging* and the *Apple Push Notification Service* will accept them.

Web push instead uses the standardized [Web Push Protocol](https://datatracker.ietf.org/doc/html/rfc8030) together with [Message Encryption for Web Push (RFC 8291)](https://datatracker.ietf.org/doc/html/rfc8291) and [Voluntary Application Server Identification (VAPID, RFC 8292)](https://datatracker.ietf.org/doc/html/rfc8292). These standards are implemented natively by every modern browser's push service (Mozilla autopush, Google FCM web endpoints, Apple Web Push, …), so **no Nextcloud push proxy is involved**. The Nextcloud server talks directly to the browser vendor's push endpoint that the browser hands out during subscription.

The trust model therefore differs from the mobile flow:

* Nextcloud server
  - Knowledge:
    + VAPID key pair (generated and stored by the Nextcloud server in app config)
    + per-subscription `endpoint`, `ua_public` (`p256dh`) key and `auth` secret (generated by the browser, sent by the client)
  - Actions:
    + Encrypts the payload for the subscription using `ua_public` + `auth` according to RFC 8291 (`aes128gcm`).
    + Signs the request with the VAPID private key (RFC 8292) so the push service can attribute it to this server.
    + Sends the encrypted notification directly to the browser's push `endpoint`.

* Browser push service (Mozilla autopush / Google FCM web endpoints / Apple Web Push / …)
  - Knowledge:
    + the VAPID public key of the server (to verify the request signature)
    + the opaque push `endpoint`
  - Actions:
    + Verifies the VAPID signature.
    + Stores and forwards the (already encrypted) payload to the browser.
    + *Note:* the push service cannot read the payload — it only ever sees ciphertext.

* Browser / Service Worker
  - Knowledge:
    + the subscription private key material (kept inside the browser, never leaves it)
  - Actions:
    + Decrypts the payload.
    + Forwards it to an open Nextcloud tab, or shows a background notification when no tab is open.

Because the payload is encrypted end-to-end between the server and the browser using keys the browser generated, the push service never learns the notification content.


## Checking the capabilities of the Nextcloud server

Web push is **opt-in** and controlled by the administrator. Whether it is available is reported in the capabilities endpoint `/ocs/v2.php/cloud/capabilities`:

```
{
  "ocs": {
    ...
    "data": {
      ...
      "capabilities": {
        ...
        "notifications": {
          "push": [
            ...
            "devices",
            "object-data",
            "delete",
            "webpush",
            "webpush-browsers"
          ]
        }
      }
    }
  }
}
```

| Capability          | Meaning                                                                                                                                                       |
|---------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `webpush`           | Web push is enabled by the administrator. Clients authenticating with an app password/token may register a subscription.                                       |
| `webpush-browsers`  | In addition, registrations from a plain browser session (no dedicated app password, e.g. the Nextcloud web UI itself) are allowed. Only present with `webpush`.|

If neither capability is present, the administrator has disabled web push and all registration endpoints will reply with `403`.

The administrator toggles these two settings in *Administration settings → Notifications* (stored as the `webpush_enabled` and `webpush_browsers_enabled` app config values).


## Subscribing at the Nextcloud server

The subscription is a four-step handshake. The browser creates a push subscription, registers it at the server, and the server confirms ownership of the subscription by sending an *activation token* through the push channel itself. Only once the client echoes that token back is the subscription marked as active and starts receiving notifications.

### 1. Fetch the server VAPID public key

```
GET /ocs/v2.php/apps/notifications/api/v2/webpush/vapid
```

#### Response

| Status code | Meaning           |
|-------------|-------------------|
| 200         | The VAPID key     |

```json
{
  "vapid": "BJ…(base64url-encoded uncompressed P-256 public key)…"
}
```

The key pair is generated on first access and persisted in the app config (`webpush_vapid_pubkey` / `webpush_vapid_privkey`). If the stored keys cannot be decrypted (for example after the instance secret changed) the server transparently regenerates them; existing subscriptions tied to the old key are then dropped by the browser via the `pushsubscriptionchange` flow below.

### 2. Create a browser push subscription

The client passes the `vapid` value as the `applicationServerKey` to the browser's `PushManager.subscribe()`. The browser returns a subscription containing:

| Field         | Description                                                                                       |
|---------------|---------------------------------------------------------------------------------------------------|
| `endpoint`    | Push service URL the server will deliver to (RFC 8030), max 765 characters.                       |
| `p256dh` key  | The subscription's uncompressed P-256 public key (RFC 8291), base64url encoded — sent as `uaPublicKey`. |
| `auth` secret | 16-byte authentication secret (RFC 8291), base64url encoded.                                      |

Some browsers (Chromium-based, and Apple following them) only permit subscriptions with `userVisibleOnly: true`. The client first tries `userVisibleOnly: false` and, on a `NotAllowedError`, retries with `userVisibleOnly: true`.

### 3. Register the subscription at the server

```
POST /ocs/v2.php/apps/notifications/api/v2/webpush

{
  "endpoint": "{{endpoint}}",
  "uaPublicKey": "{{p256dh}}",
  "auth": "{{auth}}",
  "appTypes": "all"
}
```

| Parameter     | Description                                                                                                                                                                |
|---------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `endpoint`    | Push server URL, max 765 chars (RFC 8030). Must be an `https://` URL whose host passes the server-side remote host validation.                                              |
| `uaPublicKey` | Public key of the device, uncompressed, base64url encoded (RFC 8291).                                                                                                      |
| `auth`        | Authentication tag, base64url encoded (RFC 8291).                                                                                                                          |
| `appTypes`    | Comma-separated list of app types used to filter incoming notifications. Alphanumeric values. Use `all` to receive everything; prefix a type with `-` to exclude it (e.g. `all,-spreed`). Max 256 chars. |

A subscription is uniquely identified by the pair *(user, session/app token)*: a single device (browser profile) keeps exactly one subscription. Re-registering with the same `endpoint`, `uaPublicKey` and `auth` on an already activated subscription only updates the `appTypes` filter. Any other change (or an inactive subscription) replaces the previous row and triggers a fresh activation.

#### Response

| Status code | Meaning                                                                                                |
|-------------|--------------------------------------------------------------------------------------------------------|
| 200         | A subscription was already registered and activated — only the filter was updated, no further action.  |
| 201         | New subscription registered. The server now sends an **activation token** through the push channel.    |
| 400         | Registering is not possible (see error messages below).                                                |
| 401         | Not logged in / missing permissions.                                                                   |
| 403         | Web push is disabled by the administrator (`WEBPUSH_DISABLED`).                                         |

#### Body in case of an error

In case of `400` the following `message` can appear in the body:

| Error               | Description                                                                                |
|---------------------|--------------------------------------------------------------------------------------------|
| `INVALID_P256DH`    | The `uaPublicKey` is not a valid base64url-encoded uncompressed P-256 public key.          |
| `INVALID_AUTH`      | The `auth` secret is not a valid 16-byte base64url-encoded value.                          |
| `INVALID_ENDPOINT`  | The endpoint is longer than 765 chars, not a valid URL, not `https`, or its host is not allowed by the remote host validator. |
| `TOO_MANY_APP_TYPES`| The `appTypes` string exceeds 256 characters.                                              |
| `INVALID_SESSION_TOKEN` | The authentication token of the request could not be identified.                       |

In case of `403`:

| Error              | Description                                                                                          |
|--------------------|------------------------------------------------------------------------------------------------------|
| `WEBPUSH_DISABLED` | Web push is disabled, or the request comes from a plain browser session while `webpush-browsers` is off. |

### 4. Activate the subscription

When the server replied with `201`, it immediately sends a notification through the push channel whose decrypted payload is:

```json
{
  "activationToken": "{{uuid-v4}}"
}
```

The client (service worker → web app) reads the `activationToken` and confirms it:

```
POST /ocs/v2.php/apps/notifications/api/v2/webpush/activate

{
  "activationToken": "{{activationToken}}"
}
```

This proves the client genuinely controls the subscription (it received and decrypted a message that only the real subscription could decrypt) before any real notifications are delivered.

#### Response

| Status code | Meaning                                                                       |
|-------------|-------------------------------------------------------------------------------|
| 200         | Subscription was already activated (`OK`).                                    |
| 202         | Subscription activated successfully (`CREATED`).                              |
| 400         | Activating not possible, e.g. wrong activation token (`INVALID_ACTIVATION_TOKEN`). |
| 401         | Not logged in / missing permissions.                                          |
| 403         | Web push is disabled by the administrator (`WEBPUSH_DISABLED`).               |
| 404         | No subscription found for the device (`NO_PUSH_SUBSCRIPTION`).                |


## Unsubscribing at the Nextcloud server

When the user disables notifications or the browser subscription goes away, the client removes the subscription:

```
DELETE /ocs/v2.php/apps/notifications/api/v2/webpush
```

The subscription matching the current *(user, session/app token)* is removed.

### Response

| Status code | Meaning                                              |
|-------------|------------------------------------------------------|
| 200         | No subscription existed for the device.              |
| 202         | Subscription removed successfully.                   |
| 400         | Removing not possible (`INVALID_SESSION_TOKEN`).     |
| 401         | Not logged in / missing permissions.                 |

Unlike registration, unsubscribing is **not** gated behind the admin capability — a client can always clean up its subscription even if the administrator disabled web push in the meantime.


## Subscription renewal (`pushsubscriptionchange`)

Browser push subscriptions are not permanent: the push service may invalidate them, or the server's VAPID key may have been regenerated. Two mechanisms keep the server in sync:

* When the browser fires a `pushsubscriptionchange` event, the service worker asks the web app to re-run the registration flow from step 1, which produces a fresh subscription and a new activation handshake.
* When the server delivers to an expired subscription, the push service answers with a *subscription expired* status. The server then deletes the matching row automatically (keyed by `endpoint`), so it stops trying to deliver to dead endpoints.


## Pushed notifications

The payload is encrypted by the server for the specific subscription using `aes128gcm` content encoding (RFC 8188/8291). After the browser decrypts it, the service worker receives a plain JSON object — there is **no second application-level signature/encryption layer** as in the mobile proxy flow, because the transport itself is already authenticated (VAPID) and end-to-end encrypted.

Notifications are filtered per subscription using the stored `appTypes` before sending: a notification for app `X` is delivered when `appTypes` contains `X`, or contains `all` without an explicit `-X` exclusion.

The notification `urgency` ([RFC 8030 §5.3](https://www.rfc-editor.org/rfc/rfc8030#section-5.3)) is derived from the app/type: Talk, two-factor and phonetrack notifications use `high`, everything else `normal`.

### Normal content notification

The web push payload uses the same shape as the [mobile push subject](push-v2.md#encrypted-subject-data), but it is delivered as clear (already-decrypted) JSON and may carry a longer `subject` (the payload budget is ~3000 bytes instead of ~240). If you are missing any information necessary to parse the notification, use the `nid` to fetch the full notification via the [OCS API](ocs-endpoint-v2.md).

```json
{
  "nid" : 1337,
  "app" : "spreed",
  "subject" : "Test mentioned you in a private conversation",
  "type" : "chat",
  "id" : "t0k3n"
}
```

| Attribute | Meaning                                                                                                       | Capability    |
|-----------|---------------------------------------------------------------------------------------------------------------|---------------|
| `nid`     | Numeric identifier of the notification, used to fetch more information via the [OCS API](ocs-endpoint-v2.md). | `object-data` |
| `app`     | The Nextcloud app sending the notification.                                                                   | -             |
| `subject` | The subject of the actual notification.                                                                       | -             |
| `type`    | Type of the object this notification is about.                                                                | `object-data` |
| `id`      | Identifier of the object this notification is about.                                                          | `object-data` |

### Activation notification

Sent exactly once, right after a `201` registration response, to confirm ownership of the subscription (see step 4 above). It is not shown to the user.

```json
{
  "activationToken" : "{{uuid-v4}}"
}
```

### Silent delete notification (single)

These notifications should not be shown to the user. Instead, delete pending notifications for the respective id.

```json
{
  "nid" : 1337,
  "delete" : true
}
```

| Attribute | Meaning                                          | Capability    |
|-----------|--------------------------------------------------|---------------|
| `nid`     | Numeric identifier of the notification to delete | `object-data` |
| `delete`  | Delete all notifications related to `nid`        | `delete`      |

### Silent delete notifications (multiple)

```json
{
  "nids" : [1337],
  "delete-multiple" : true
}
```

| Attribute         | Meaning                                                  | Capability    |
|-------------------|----------------------------------------------------------|---------------|
| `nids`            | Numeric identifiers of the notifications to delete (batched in groups of up to 10) | `object-data` |
| `delete-multiple` | Delete all notifications related to `nids`               | `delete`      |

### Silent delete notification (all)

```json
{
  "delete-all" : true
}
```

| Attribute    | Meaning                                          | Capability |
|--------------|--------------------------------------------------|------------|
| `delete-all` | Delete all notifications related to this account | `delete`   |


## Handling notifications in the client

The app ships a service worker (registered from `/apps/notifications/service-worker.js`) that handles the browser `push` event:

1. If a Nextcloud tab is open, the decrypted payload is forwarded to it via `postMessage` (`{ type: 'push', content }`) and the in-page UI handles it. If the payload carries an `activationToken`, the page performs the activation request instead.
2. If no tab is open and the payload has a `subject`, the service worker shows a background notification (first line of the subject becomes the title, the rest the body).
3. If there is no displayable content (e.g. a delete payload arriving with no open tab) and the browser requires `userVisibleOnly`, a short-lived silent placeholder notification is shown so the browser does not unregister the subscription for staying silent.

On `pushsubscriptionchange` the service worker asks the page to re-register (`{ type: 'pushEndpoint' }`), driving the renewal flow described above.
